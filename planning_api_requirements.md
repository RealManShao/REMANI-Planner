# 路径规划任务接口需求分析报告

## 1. 现状分析

根据《项目API要求》文档，目前上层系统提供的接口（主要集中在运动学与动力学层）包含：

- **模型与实例管理**：注册模型、配置实例、标定参数。
- **正向运动学与动力学**：正向运动学 (FK)、笛卡尔速度/加速度计算、雅可比矩阵计算、静态力矩计算。
- **安全性与工具**：奇异性检测、构型识别、关节限位检查、矩阵变换辅助计算。

**当前存在的主要问题：**
目前的 API 仅提供了机器人本体的数学模型计算和基本状态定义，**缺乏对外部环境的感知、任务目标的定义、运动控制的闭环以及逆向运动学（IK）的实现**。仅依靠现有接口，规划器只能做孤立的数学运算，无法实现“从起点避障到达终点”的路径规划核心任务。

***

## 2. 路径规划任务还需补充的接口需求

为了完成复合移动机械臂的全身路径规划任务，上层系统还需要在以下几个分类中提供支持接口：

### 2.1 逆向运动学（IK）接口（全身协同）

由于移动机械臂存在高度冗余，传统的单一机械臂 IK 无法直接适用，必须提供支持全身协调的运动学求解。

- **`computeWholeBodyInverseKinematics`** **(计算全身逆向运动学)**
  - **功能**：将目标末端笛卡尔位姿转换为全身构型（包含底盘位姿和机械臂关节角）。
  - **参数**：末端目标位姿、底盘参考位置（可选，用于打破冗余）、当前全身状态（作为迭代初值）。
  - **必要性**：这是移动操作（Mobile Manipulation）的核心。

### 2.2 环境感知与碰撞检测接口（静态环境抽象层）

**由于本任务的场景为封闭、固定的静态环境，规划器不需要进行运行时的实时动态避障，只需在规划开始前获取一次全局环境信息即可。**

- **`getStaticEnvironmentMap`** **(获取静态环境地图/障碍物)**
  - **功能**：在规划初始化阶段，获取工作空间内的静态障碍物拓扑信息（例如 3D 体素网格 Voxel Grid、OctoMap 或 ESDF 地图）。
  - **调用频率**：单次调用（初始化时）。
- **`checkWholeBodyCollision`** **(全身碰撞检测)**
  - **功能**：输入一组全身状态（底盘位姿 + 关节角）或一条离散轨迹，返回是否与静态环境或机器人自身发生碰撞。
  - **必要性**：虽然环境是静态的，但规划算法（如 A\*, RRT）在内部搜索树扩展时，依然需要对每一个采样点进行自碰撞和环境碰撞的几何检测。如果环境地图较大，建议上层提供基于 C++ 的低延迟回调接口。

### 2.3 初始状态获取与目标下发接口

在静态封闭场景下，规划器只需在任务开始时获取一次机器人的初始状态。

- **`getInitialWholeBodyState`** **(获取全身初始状态)**
  - **功能**：在接到规划任务时，获取机器人当前的实际全身状态（底盘里程计位姿 $x, y, \theta$ 以及机械臂的关节位置）。
  - **调用频率**：每次任务开始前调用一次。
- **`setPlanningTask`** **(下发规划任务)**
  - **功能**：接收上层业务下发的规划目标（通常是末端执行器在全局坐标系下的 6D 目标位姿，或者带约束的抓取区域）。

### 2.4 轨迹下发与反馈接口（运动执行层）

规划出轨迹后，需要将完整的轨迹下发给底层控制器执行。

- **`executeWholeBodyTrajectory`** **(下发全身轨迹)**
  - **功能**：将规划生成的轨迹下发给控制器。**该轨迹必须是时空同步的（Spatial-temporal）**，即在同一个时间戳下，同时包含底盘的目标速度指令和机械臂的关节角/速度指令，以确保全身协调运动。
- **`getExecutionStatus`** **(获取执行状态)**
  - **功能**：由于环境是静态的，规划器不需要高频轮询，只需在下发轨迹后，偶尔查询或等待回调，确认轨迹是否已成功执行完毕即可。

### 2.5 坐标系与 TF 树管理（坐标系管理层）

- **`lookupTransform`** **(查询坐标系变换)**
  - **功能**：查询当前时刻任意两个坐标系（如全局地图 `map` 到底盘 `base_link`，或底盘 `base_link` 到末端 `tool0`）之间的动态变换关系。
  - **必要性**：移动机械臂在运动过程中，底盘基座位置不断变化，必须依赖动态坐标树来对齐目标点和环境障碍物。

## 3. 现有接口现存问题与优化建议

结合已有的接口，针对**复合移动机械臂**的特性，提出以下对接问题和优化建议：

1. **完全缺失移动底盘的模型定义与约束**
   - **问题**：现有的 `registerKinematicModel` 接口（支持 MDH, POE, URDF 等）完全是为固定基座机械臂设计的，没有底盘（非完整约束系统，Non-holonomic）的运动学和动力学参数定义（如最大线速度、最大角速度、底盘形状等）。
   - **建议**：在模型定义中增加对移动底盘（Mobile Base）的抽象，或者增加一种特殊的 `MobileManipulator` 模型类型，允许将移动底盘与机械臂模型进行绑定组合。
2. **缺失全身 IK 接口**
   - **问题**：当前缺少逆向运动学接口，更缺乏针对高冗余度移动机械臂的全身 IK（Whole-body IK）求解器。
   - **建议**：补充 `computeWholeBodyInverseKinematics` 接口。对于冗余系统，该接口应支持权重配置（例如更倾向于移动底盘，还是更倾向于伸展机械臂）。
3. **接口响应的实时性要求**
   - **问题**：碰撞检测和 IK 在路径规划中会被成千上万次高频调用。如果是基于 HTTP/RESTful 的 JSON 交互，网络和序列化开销将导致规划耗时极长。
   - **建议**：对于高频调用的辅助计算接口（碰撞检测、运动学计算），强烈建议在上层系统提供 **gRPC** 或本地库链接 (C++ SDK) 方式。
4. **动力学约束与时空优化支持**
   - **问题**：目前提供了软限位检查。但在全身规划中，为了实现安全和敏捷的运动，还需要考虑底盘的动力学约束（加速度限位）和机械臂的动力学约束。
   - **建议**：在模型详情接口中，明确暴露全身各自由度（包括底盘）的最大速度、最大加速度、以及动力学边界参数，供轨迹优化算法（如 MINCO 轨迹表示法）使用。
